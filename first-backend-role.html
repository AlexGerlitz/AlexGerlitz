A bounded first backend task is the right screen.

Team Fit

  • Russian-speaking product/backend team where the first result is a small reviewed backend ticket.
  • English-first async team where the review path is issues, pull requests, tests, docs, and handoff notes.

Good First Task

Give me a bounded backend task with a real workflow, expected behavior, review path, and enough access to test safely. The output should be code, tests or smoke checks, docs, and a clear next slice.

Wrong First Screen

Do not evaluate me by asking me to own a large architecture alone on day one. The stronger screen is one small backend responsibility inside team review.

First-Role Questions

Can I start small?Give me one endpoint, model field, admin queue step, integration mapping, test gap, or runbook gap.
Can I work inside review?The best first slice has a small diff, tests or smoke checks, docs, and a reviewer-visible handoff.
Can I handle backend state?The Autoschool proof shows request intake, validated payload, database-style record, admin queue row, status update, and audit context.
Can I protect business data?Public proof uses synthetic data only and keeps live admin names, contact details, IDs, URLs, logs, dumps, and credentials out of GitHub.
Can I grow into broader ownership?The route connects first tasks to FastAPI, PostgreSQL, OpenAPI, Docker, CI, integrations, admin workflows, and runbooks.

Best First Tasks

  • Add or fix a FastAPI endpoint with request validation and response shape.
  • Add a PostgreSQL model field, status transition, migration note, and small test.
  • Turn one Telegram/chat request into a normalized backend payload and admin queue row.
  • Add an admin filter, status action, or operator handoff note.
  • Map one CRM/API integration payload with validation and error handling.
  • Add a smoke check, CI check, log note, or runbook section for an existing workflow.
  • Write the issue-to-proof handoff: what changed, how to test it, and what remains risky.

Public Evidence Boundary

This first-role screen uses public-safe proof only. It does not include live admin screenshots, real learner names, contact details, Telegram IDs, chat IDs, private URLs, raw logs, credentials, database dumps, or private repository code.
