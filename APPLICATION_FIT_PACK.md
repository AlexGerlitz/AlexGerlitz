deeper review path. It is written for honest matching: exact role titles, exact stack words, one
first useful result, and the public evidence that supports the claim.

## Market-Backed Priority Order

I use this order for applications:

1. Python backend / internal tools - primary first-job lane where FastAPI, PostgreSQL, SQL,
   REST API, OpenAPI, pytest, Docker Compose, GitHub Actions, admin workflow, and handoff
   evidence matches the strongest public work samples.
2. QA automation / API verification with Python - practical entry lane for Quality Assurance
   Automation Engineer, QA Automation Python, API Testing, Test Automation Engineer, and
   Support Engineer with Python roles where I can prove APIs, SQL/data state, logs, smoke
   checks, and repeatable verification.
3. Integration / API / CRM - strong business-workflow lane when the role needs field mapping,
   adapters, webhooks, retries, audit logs, privacy boundaries, and a small working sync or
   intake-to-admin path.
4. AI workflow / RAG - differentiator when the job already has backend, integration, or QA
   ownership and AI is part of a verifiable workflow instead of a vague prompt/content task.

Docker Compose and GitHub Actions support the review path; I do not lead with senior DevOps,
Kubernetes, Terraform, or cloud platform ownership as the main claim.

Review path: Delivery Capability -> First Backend Role Fit -> Backend Role Fit -> Autoschool

Backend Review Pack.

Hiring Signal Brief.

## Application Message Shape

I keep first messages short and role-specific:

1. I name the role lane I match.
2. I name the first useful result I can own.
3. I name the exact stack overlap.
4. I attach one review path instead of many random links.
5. I ask for the first task shape, review owner, and success check.

For copy-ready vacancy replies and recruiter-message variants, I use
[Application Outreach Pack](https://alexgerlitz.github.io/AlexGerlitz/application-outreach-pack.html).
It turns this fit map into first-person messages, ATS keyword bundles, vacancy triage, truthful
Docker/CI and AI boundaries, privacy limits, and lane-specific review paths.

Example:

```text
Hi, I am interested in this remote Python/backend automation role. The strongest match is internal
tools plus API workflow ownership: Python, FastAPI, PostgreSQL/SQL, REST API, Docker Compose,
GitHub Actions, tests or smoke checks, logs, docs, and handoff. My fastest review path is Delivery
Capability -> First Backend Role Fit -> Autoschool Intake/Admin work sample. I can start with one
bounded workflow slice: input, validation, state, output, verification, and handoff.
```

## Not My First Application Lane

I should not lead with Kubernetes, Terraform, AWS platform ownership, highload infrastructure,
onsite-only roles, pure prompt/content tasks, isolated brochure websites, generic ecommerce/mobile
apps without backend or integration ownership, or undefined work where success cannot be verified.
