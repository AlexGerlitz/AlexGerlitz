## Category Strategy

LinkedIn categories are broad routing labels, not the actual offer. The service page is triaged by
system ownership: backend/platform, AI workflow, integration, DevOps, data, internal operations, and
handoff quality. Categories that mostly route isolated websites, student assignments, standalone
games, or generic mobile/ecommerce requests should be removed or treated as not a fit.

Keep:

- Cloud Application Development
- Custom Software Development
- Information Management
- IT Consulting

Remove:

- Web Development
- Application Development

Primary add set:

- Cloud Management
- Database Development
- Data Reporting
- Business Analytics
- Software Testing
- Cybersecurity

Fallback only if one primary category is unavailable:

- Technical Support

Do not add:

- Mobile Application Development
- broad web-only categories
- categories that mainly route design, content, marketing, simple site refreshes, student work, or
  generic low-level support

## Request Triage Rule

Good fit:

- backend-owned AI workflow, RAG, transcript/call/document analysis, approval flow, CRM action, or
  internal operator workflow;
- CRM/ERP/1C/banking/accounting/API/database integration with contracts, retries, logs, and
  rollout notes;
- FastAPI/PostgreSQL backend, admin workflow, audit trail, worker/outbox, tests, Docker, CI, and
  runbook;
- DevOps/recovery/hardening work with deploy, logs, health checks, smoke tests, backup/restore, or
  rollback.

Not a fit unless there is a real backend/integration system to own:

- isolated website refresh;
- student or course assignment;
- standalone game clone;
- generic mobile/ecommerce app;
- vague request with no system boundary and no verifiable success condition.
