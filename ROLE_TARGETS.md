## Priority Order For First Job Search

I use this order when choosing where to apply first:

1. Python backend / internal tools - strongest public evidence for first-job conversations: FastAPI,
   PostgreSQL, SQL, REST API, OpenAPI, pytest, Docker Compose, GitHub Actions, admin workflows,
   and handoff notes.
2. QA automation / API verification with Python - practical entry lane for Quality Assurance
   Automation Engineer, QA Automation Python, API Testing, and Test Automation Engineer roles
   where backend workflow verification, SQL/data checks, logs, and smoke routes matter.
3. Integration / API / CRM - strong business lane when the role needs adapters, webhooks, field
   mapping, privacy boundaries, retries, audit logs, and a small working sync or intake/admin path.
4. AI workflow / RAG - differentiator when the AI work is attached to backend, integration, QA,
   data quality, approvals, and supportable workflow ownership.

Docker/CI is supporting evidence for reliability and handoff, not the main senior DevOps claim.

## First 30 Days I Can Own

For a role, I am easiest to evaluate through a real workflow, not a generic exercise.

| Area | First-month ownership I can take |
| --- | --- |
| Existing backend | Map the current flows, identify risky boundaries, add tests/smoke checks, improve docs, and ship a small feature or workflow slice. |
| AI automation | Pick one document/transcript/ticket/operator workflow, build ingestion/retrieval or routing, add approval states, and prove it with repeatable tests. |
| AI sales/support operator | Build the DriveDesk AI Operator slice: transcript/lead ingestion, RAG retrieval, analysis JSON, Telegram approval, CRM sandbox adapter, audit, retries, screenshots, and runbook. |
| QA/API verification | Add pytest/API smoke checks, SQL/data checks, issue notes, logs, and a clear pass/fail route for one business workflow. |
| Integration work | Define source/target contracts, map data fields, add validation and logging, build a first sync/import path, and document rollout risks. |
| Docker/CI / recovery | Add or tighten Docker/CI/health/logging/smoke checks, document backup/restore or rollback, and leave a runbook. |
| Internal tools | Turn one manual workflow into records, roles, admin actions, notifications/tasks, and a deployable backend surface. |

## What I Do Not Need To Be Productive

I do not need a perfectly specified task before starting. A useful starting point is enough:

- a messy workflow;
- a current tool or database;
- a failing or manual process;
- an API boundary;
- a deadline and one success condition.

I will turn that into assumptions, risks, a small working slice, and a verification plan.
