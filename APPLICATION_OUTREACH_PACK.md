# Application Outreach Pack

Public route: https://alexgerlitz.github.io/AlexGerlitz/application-outreach-pack.html

I use this pack when I need to answer a vacancy, recruiter message, or application form with one clear role lane, one work sample, and a scope boundary. It keeps outreach from becoming a generic
intro or an overclaim.

## Best Use

Use this when the role is remote and asks for one of these surfaces:

- Python backend, internal tools, admin workflows, FastAPI, PostgreSQL, REST API, OpenAPI, pytest,
  Docker Compose, GitHub Actions, docs, or handoff.
- QA automation, API testing, Python checks, SQL/data checks, smoke checks, logs, and issue notes.
- Integration work around CRM, ERP, APIs, webhooks, Telegram, 1C-facing workflows, banking-facing
  workflows, audit, retries, idempotency, and privacy boundaries.
- AI workflow work where retrieval, structured output, approvals, logs, CRM-safe handoff, and
  supportable runbooks matter more than prompt-only work.

Do not use it as the main route for senior DevOps, Kubernetes, Terraform, cloud platform ownership,
onsite-only roles, pure prompt/content work, isolated static sites, generic mobile/ecommerce apps,
or vague work where the first result cannot be tested.

## Market-Backed Priority

Checked on 2026-07-01.

  Python, support-with-Python, and integration roles where the compensation band is clear before a

  smoke checks, runbooks, and handoff visible, but do not make geography the public ask.

  and work samples aligned around the same role lanes.

## Fast Apply Messages

Use one short note per application. Pick the closest lane, keep one proof route, and do not stack
every project link into the first message. Write in the vacancy language: Russian for HH/Telegram
Russian-language roles, English for English-language roles, with the same proof links and scope
boundary.

## Scope Boundary

I can work with Docker Compose, GitHub Actions, health checks, smoke checks, logs, backup notes,
runbooks, and handoff docs as part of backend/internal-tools delivery. I do not lead with senior
DevOps, Kubernetes, Terraform, or cloud platform ownership as the primary claim.

I use AI tooling to compress discovery, implementation, debugging, docs, and review, while
architecture, state boundaries, privacy checks, tests, logs, deployment notes, runbooks, and shipped
quality stay my responsibility.

I do not attach live admin captures, real names, phone numbers, chat IDs, private routes, logs,
dumps, record identifiers, tokens, or credentials. Public evidence uses synthetic or redacted data.

| Signal | Apply now | Adapt first | Skip |
| --- | --- | --- | --- |
| Role lane | Python/backend, QA/API Python, integration/API/CRM, AI workflow with backend ownership | Backend role with unclear first task or broad AI wording | Senior platform, Kubernetes/Terraform-first, onsite-only, pure content |

## Screening Boundaries

### Experience Boundary

Yes. I should be evaluated by a bounded first slice and public evidence, not by vague seniority labels.
Since March 2024 I have supported real business digital operations and turned that context into
public-safe work samples: Telegram request intake, backend validation, database state, admin queue,
operator workflow, AI/RAG workflow, integration boundaries, tests, smoke checks, docs, and handoff.

### AI Ownership Boundary

Yes. I use AI tooling to move faster through discovery, implementation, debugging, docs, and review,
but I keep the engineering responsibility on my side: architecture, state, privacy, tests, logs,
runbooks, handoff, and shipped quality.

### Docker/CI Boundary

Yes for backend delivery support: Docker Compose, GitHub Actions, smoke checks, health checks, logs,
release notes, backup notes, and runbooks. I do not sell this as senior DevOps or platform ownership.

### First Task Shape

Ask for one bounded workflow slice with input, state, output, verification, and handoff. Good examples:
one API check, one admin queue path, one adapter contract, one RAG evaluation path, one retry/audit
path, or one smoke-checked backend route.

## Application Loop

1. Pick one lane: Python backend/internal tools, QA/API verification, integration/API/CRM, or AI workflow.
2. Use one work sample, not a pile of links.
3. State one first useful result with input, state, output, verification, and handoff.
4. Keep Docker/CI and AI as supporting evidence unless the role clearly asks for them.
5. Ask for the first task shape, review owner, success check, timeline, and compensation band if shareable.
